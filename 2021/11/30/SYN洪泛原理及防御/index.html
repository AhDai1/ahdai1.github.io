SYN洪泛原理及防御

Posted on 2021-11-30 | In network | | post.views:
Words count in article: 3.3k 字 | Reading time ≈ 12 分钟

SYN洪泛

TCP SYN洪利用TCP协议三次握手。攻击者发送TCP SYN，SYN是TCP三次握手中的第一次握手，而当服务器返回ACK后，该攻击者就不对其进行再确认，那这个TCP连接就处于挂起状态，也就是所谓的半连接状态，服务器收不到再确认的话，还会重复发送ACK给攻击者。这样更加会浪费服务器的资源。攻击者就对服务器发送非常大量的这种TCP连接，由于每一个都没法完成三次握手，所以在服务器上，这些TCP连接会因为挂起状态而消耗CPU和内存，最后服务器可能宕机，就无法为正常用户提供服务了。

SYN原理

从SYN洪泛的概述来看，主要是利用TCP的第一次握手。上面提到了半连接状态，我们先来了解一下什么是半连接，什么是全连接。

首先服务端在调用 listen() 后就等待客户端的连接请求，这时服务端监听端口处于 LISTEN 状态。当客户端发起了连接请求，也就是向服务端发送了 SYN 请求，服务器收到 SYN 请求后，端口会转变到 SYN_RECV 状态，这时候服务端内核就会将该连接储存到半连接队列，并向客户端响应 SYN + ACK，接着客户端会回应 ACK，服务端收到第三次握手的 ACK 后，端口状态变成ESTABLISHED，内核会把连接从半连接队列中移除，然后创建新的完全的连接，并将其添加到全连接队列中，等待服务端调用 accept() 时把连接取出来。

SYN洪泛如何实现

服务端内核中有两个队列用来维护连接，而SYN洪泛正是利用了服务端的半连接队列。正常的客户端收到服务端回送的 SYN + ACK 后会回复 ACK，而恶意客户端在发送完 SYN 请求后而不对服务端响应 ACK就结束握手，而此时服务端并不知道正常的 ACK 是由于网络阻塞导致的延缓还是恶意连接的不回应 ACK，所以服务端必须进行等待。

每一个连接都会消耗服务器的资源，而如果成千上万的恶意程序对服务器发送请求而不回应 ACK，服务端就会维护很多半连接状态的连接，如果短时间内服务器资源被这些半连接占满，那么正常的业务请求在这期间就会得不到响应，处于等待状态。

这种恶意程序发送 SYN 请求而不回应 ACK 就称为 SYN 洪泛（SYN flood）攻击，也就是DoS（Denial of Service拒绝服务）中的一种。

SYN洪泛的种类

在前面我们了解了 SYN 洪泛攻击的原理，而 SYN 洪泛攻击还分直接攻击、欺骗式攻击与分布式攻击等等。

直接攻击

如果攻击者用他们自己没有经过伪装的IP地址快速发送 SYN 请求，这就是直接攻击。这种攻击要想奏效，攻击方还需要阻止自己内核相应 SYN + ACK，因为任何 ACK、RST或 ICMP 等都将让服务器跳过 SYN_RECV 状态。

攻击者可以设置防火墙规则来实现阻止发送回应，例如让防火墙阻止一切除了 SYN 要到达服务器的数据包，或让防火墙阻止一切进来的包，使 SYN + ACK 在到达本地 TCP 处理程序之前就被丢弃了。

欺骗式攻击

SYN洪泛攻击还可以进行 IP 地址的欺骗，也就是将用有效的 IP 和 TCP 报文头去替换和重新生成原始的 IP 报文，这就是欺骗式攻击。

对于欺骗式攻击，首先要考虑的就是选择地址，要想攻击成功，位于这个 IP 地址的主机不能响应任何 SYN + ACK。但很多伪装的 IP 地址上并没有主机，所以就根本不可能会响应。

分布式攻击

对于只伪装成另一个 IP 地址的攻击者，服务端可以通过一些策略或和 ISP 间的配合，找到真实的 IP 地址。但是攻击者如果运用网络中主机数量上的优势来发动分布式攻击，SYN 洪泛将更难以阻止。

本以为是个非常简单的实验，跌跌撞撞终于做完了。

实验起初我在本机利用 hping3 进行洪范攻击，但服务端怎么都不会出现 SYN_RECV 状态的连接，还会很离奇的出现 ESTABLISHED 状态的连接，令我百思不得其解。

一开始我在本地进行抓包，发现也没有什么问题，发出的只有 SYN 包，并没有向服务端回应 ACK ，而服务端却异常出现 ESTABLISHED。接下来我去服务器进行抓包，却离奇的发现居然能收到我本机发送的 ACK 包。（我的猜测可能是内核中有某种逻辑吧，目前我也是没有搞清楚）

接下来知道问题就是服务端会收到客户端的 ACK，那我就直接在服务端将本机客户端的所有 ACK 包全部拦截丢弃，这下总不会出现 ESTABLISHED了吧，果然现在就成功出现了很多很多 SYN_RECV 状态的连接。

SYN洪泛的措施

对于 SYN 洪泛有许多防御措施，例如增大服务器半连接队列、开启 tcp_syncookies 或减少 SYN + ACK 重传次数。

增大半连接队列

SYN 洪泛的基本原理是依赖于服务半连接队列长度的溢出，因此一个显而易见的解决方案就是增大服务端的半连接队列长度。但这种方式并不能够作为防范 SYN 洪泛的有效方法，因为即使半连接队列设置的再大，攻击者也能生成比半连接队列还大的多的数据包向服务端请求服务。

接下来分析一下半连接队列的长度与哪些因素有关

tcp_max_syn_backlog

首先来看一个参数，tcp_max_syn_backlog 定义了处于 SYN_RECV 状态的最大 TCP 连接数，也就是半连接队列的最大值。但 TCP 的半连接队列大小并不只是由这个参数决定的，我们往下看。

半连接队列的大小

半连接队列的大小不仅仅是一个参数这么简单，可以从Linux 2.6.39内核源码来看TCP第一次握手后的处理逻辑：

从源码看出，处于 SYN_RECV 状态的连接可能满足三种条件而被丢弃。
1、当半连接队列满了，并且没有开启 tcp_syncookies，则会丢弃。
2、当全连接队列满了，并且没有重传 SYN+ACK 的连接请求大于 1 个，则会丢弃。
3、如果没有开启 tcp_syncookies，并且 max_syn_backlog 减去当前半连接队列的长度小于 ( max_syn_backlog >> 2)，则会丢弃。

所以修改 tcp_max_syn_backlog 的值是增大半连接队列的一个方法，但是不是决定性因素，当然影响半连接队列的因素也不仅仅只有上面的内容，还有许多因素值得去学习。

syncookies 原理

SYN cookies 是对 TCP 服务端三次握手的一次对于 SYN 洪泛的优化，在 TCP 收到 SYN 请求并返回 SYN + ACK 时，不分配一个专门的数据区，而是根据这个 SYN 请求计算出一个 cookies 值。在收到客户端回应的 ACK 时，服务端根据 cookies 值检查这个 ACK 的合法性。如果合法再分配专门的数据区来处理连接。

这样在连接还没建立成功时就不会造成服务端的资源被半连接耗尽。

设置 syncookies 值

syncookies 参数的值有三个

  • 0，代表关闭该功能

  • 1，代表仅当 SYN 半连接队列满了再启用

  • 2，代表无条件开启功能

减少 SYN + ACK 重传次数

原理

当服务端收到 SYN 攻击时，就会有大量处于 SYN_RECV 状态的 TCP 连接，处于这个状态的 TCP 会重传 SYN+ACK，当重传超过次数超过上限后，就会断开连接。

针对 SYN 洪泛的攻击场景，我们可以减少 SYN+ACK 的重传次数，以加快 SYN_RECV 状态的 TCP 断开连接。

但这种方案在减少 SYN_RECV 状态时间和攻击者发包率仅仅是一个线性关系。同样也有可能造成正常的客户端因网络阻塞而导致的 ACK 滞后，服务端如果在这种情况下丢弃连接，就会造成网络资源的浪费。
